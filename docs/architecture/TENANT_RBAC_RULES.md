# TENANT_RBAC_RULES

How tenant context and RBAC work. Read before adding roles, permissions, or guards.

## Tenant context resolution

- Public routes: no tenant.
- Platform routes: no tenant; require platform permission / `isPlatformSuperAdmin`.
- Tenant routes: tenant REQUIRED and always validated server-side.

Source of `tenantId` (in priority): authenticated session selection → `X-Tenant-Id`
header → (future) subdomain → (future) custom domain. **Backend always validates.**

`TenantContextInterceptor` (`common/interceptors/tenant-context.interceptor.ts`):
- Reads `X-Tenant-Id`, checks `UserTenantMembership.isActive` → else `Forbidden`.
- Reads `X-Branch-Id`, checks `UserBranchMembership.isActive` → else `Forbidden`.
- Populates `request.tenantContext = { tenantId, branchId?, userId }`.
- Marks `isPlatformSuperAdmin` when applicable.

Access it in handlers via `@TenantCtx() ctx: TenantContext`.

Required request context to carry through: userId, tenantId (where applicable),
branchId (where applicable), roles, permissions, traceId, ipAddress, userAgent.

## Roles (seed these)

Platform Super Admin · Platform Support · Tenant Owner · Tenant Admin ·
Branch Manager · Sales Executive · Ticketing Officer · Visa Officer ·
Finance Officer · HR Manager · Team Lead · Viewer.

RBAC is tenant-aware AND branch-aware: `UserRoleAssignment` is keyed by
`userId + tenantId (+ branchId?)`. The `PermissionsGuard` collects permissions from
all matching assignments and requires ALL `@RequirePermissions(...)` to be present.

## Permission naming — IMPORTANT divergence

- **Product spec** asks for `module:action` (e.g. `leads:create`).
- **Current code** uses `MODULE_ACTION` (e.g. `LEAD_CREATE`), generated by
  `@travelo/permissions` as `Modules × Actions`:
  - Modules: TENANT, BRANCH, USER, ROLE, PERMISSION, LEAD, CLIENT, FOLLOW_UP,
    QUOTATION, BOOKING, TICKET, INVOICE, RECEIPT, PAYMENT, EXPENSE, LEDGER,
    EMPLOYEE, LEAVE, ATTENDANCE, PERFORMANCE, DOCUMENT, SETTINGS, AUDIT_LOG,
    REPORT, DASHBOARD, NOTIFICATION, MASTER_DATA
  - Actions: CREATE, READ, UPDATE, DELETE, MANAGE

**Rule until migrated:** keep using `MODULE_ACTION` in `@RequirePermissions` and the
seed. A switch to `module:action` (if adopted) must be a deliberate, migrated change
that updates the catalog, guard, seed, and all decorators together. Do not mix forms.

## Enforcement checklist per endpoint

1. `@UseGuards(JwtAuthGuard, TenantGuard, PermissionsGuard)` on the controller.
2. `@RequirePermissions('MODULE_ACTION')` on the handler.
3. Scope every query by `ctx.tenantId` (and `ctx.branchId` when branch-scoped).
4. Never grant platform bypass inside tenant services beyond the guard's
   `isPlatformSuperAdmin` check.

## Sensitive actions

Payment verification, refund approval, ticket void/reissue, invoice void, ledger
adjustment, HR/payroll approval, role/permission change, user deactivation, data
export, tenant suspension, destructive delete → require explicit permission + audit
log + reason. Design for dual-approval and OTP/re-auth later. See `SECURITY_RULES.md`.

## Never

- Trust frontend `tenantId` without membership validation.
- Create tenant business records without `tenantId`.
- Use branch-only isolation.
- Ship a global-only role model for tenant data.
